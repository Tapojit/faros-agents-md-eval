# AGENTS.md Eval Harness

Measures whether a generated AGENTS.md actually helps a coding agent complete tasks more efficiently. Spawns independent agents on a task with and without the AGENTS.md, measures what happens, iterates.

Follows the eval framework from the [Anthropic evals blog](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents) and is informed by the ETH Zurich finding (arXiv 2602.11988) that verbose auto-generated AGENTS.md files often *hurt* performance.

## Architecture: Three Roles

This harness has three distinct roles. Conflating them is the #1 source of eval contamination.

| Role | Model | Job | MUST NOT |
|------|-------|-----|----------|
| **Evaluator** | (scripts / graders) | Orchestrate, grade, record | Execute the build task, leak knowledge to agents |
| **Agent** | Haiku (weak) | Execute the build task cold | Access evaluator context, prior trial results |
| **Generator** | Sonnet (strong) | Analyze transcripts, produce AGENTS.md | Execute the build task |

**Why a weak agent?** The whole point is measuring whether AGENTS.md helps an agent that *needs* the help. Haiku is more likely to struggle without guidance, which makes the AGENTS.md delta visible. If both conditions pass 100%, the eval is useless.

**Why a strong generator?** The AGENTS.md generator needs to read transcripts, identify failure patterns, and write precise instructions. This requires stronger reasoning than the build task itself.

## Execution Engine: `claude -p`

All agent trials run via `claude -p --output-format stream-json --verbose`. This produces a full JSONL transcript of every message, tool call, tool result, and thinking block. The transcript is the source of truth for all metrics.

**Why not the Agent tool?** The Agent tool returns a summary — we lose every tool call, every error message, every retry attempt. Without full transcripts, we can't analyze where agents struggle, which means automated AGENTS.md generation is impossible.

**Why not `--worktree`?** Claude's `--worktree` flag auto-creates a worktree, but we need to inject files (AGENTS.md, build_env.sh) *before* the agent starts and run graders *after* it finishes. Manual worktree management gives us this control.

JSONL transcripts enable:
- Deterministic metrics extraction (same transcript → same numbers, always)
- Tool call sequence analysis (what did the agent try, in what order?)
- Error pattern detection (what failed, and how did the agent recover?)
- Cost tracking (exact token counts and USD from the API)

## Trial Isolation

Each trial agent is fully sandboxed to prevent cross-contamination:

- **No `--add-dir`:** The agent runs `cd`'d into its worktree. It cannot see the workspace, other trials, evaluator scripts, or metrics.
- **`--disable-slash-commands`:** Prevents the agent from loading skills or commands from the orchestrator session.
- **Self-contained worktrees:** `setup_worktrees.sh` copies AGENTS.md and build_env.sh INTO each worktree. The agent finds everything locally — no workspace-level path references in prompts.
- **Doc stripping:** `strip_build_hints.sh` runs once during `/eval-init`, removing `README*`, `Doc/`, `.github/CONTRIBUTING*`, `Mac/README*`, `PCbuild/*.txt`, `Tools/README*` and committing the deletion. Every worktree inherits the clean state automatically. The eval measures AGENTS.md quality, not the agent's ability to find in-tree docs.
- **`--tools "Bash,Read,Grep"`:** Agent gets only the tools it needs. No Write, no Edit, no Agent.
- **`--no-session-persistence`:** Trial sessions don't save to history, preventing cross-session leakage.

## Baseline: /init-Generated AGENTS.md

The baseline condition uses an AGENTS.md generated by running `claude /init` on the repo — the standard out-of-box auto-generated file. This is a much more meaningful comparison than "no AGENTS.md at all":

- **Old approach:** baseline = no AGENTS.md → too easy to beat, proves nothing useful
- **New approach:** baseline = /init AGENTS.md → hill-climbed version must outperform the standard auto-generated one

Both conditions get the same prompt structure (both tell the agent to read AGENTS.md). The only difference is which AGENTS.md is placed in the worktree.

## Information-Asymmetric Prompts

The asymmetry is now between the AGENTS.md *content*, not its presence:

- **Baseline:** The agent gets the /init-generated AGENTS.md — generic guidance that `claude /init` would produce for any repo.
- **Improved:** Same prompt, but with a hill-climbed AGENTS.md that encodes specific knowledge from transcript analysis (non-obvious build context, exact flags, known gotchas).

## The Hill-Climbing Loop

```
0. INIT: Generate baseline AGENTS.md via `claude /init` on the repo
1. BASELINE: Run agents (Haiku) with /init AGENTS.md → get transcripts
2. GENERATE: Sonnet analyzes baseline transcripts → produces first improved AGENTS.md
3. IMPROVED: Run agents (Haiku) with improved AGENTS.md → get transcripts
4. GRADE: Deterministic graders (did it build? do imports work?)
5. COMPARE: Metrics table (pass@1, tokens, tool calls, wall clock, cost)
6. SHRINK: Sonnet analyzes improved transcripts → smaller AGENTS.md
7. REPEAT: Until AGENTS.md size plateaus or pass@1 drops
```

Key: the AGENTS.md is NEVER hand-written. It's always generated — either by `/init` (baseline) or by Sonnet from transcript analysis (improved).

```bash
# One command does everything:
bash "$SKILL_DIR/scripts/hill_climb.sh" workspace 3 haiku sonnet 3 30

```

Or step by step:
```
/eval-init            # Set up repo, deps, workspace, generate /init baseline
/eval-run baseline    # Run baseline trials (with /init AGENTS.md)
/eval-run improved    # Generate improved AGENTS.md + run improved trials
/eval-metrics         # Display comparison tables
/agents-md-improve    # Hill-climb: shrink AGENTS.md, re-test
/eval-regress         # Run regression after improvements
```

## Scripts

All scripts live in `$SKILL_DIR/scripts/`. Run with `bash` or `python3`.

### Core pipeline (v1.0.0)
- **`run_trial.sh`** — Atomic: runs one trial via `claude -p`, saves transcript + result to results dir
- **`extract_metrics.py`** — Deterministic: parses `transcript.jsonl` → metrics (JSON, CSV, timing.json)
- **`run_eval.sh`** — End-to-end: worktrees → prompts → trials → grade → extract → aggregate → viewer
- **`generate_agents_md.py`** — Analyze transcripts from results/ → generate/improve AGENTS.md via `claude -p`
- **`hill_climb.sh`** — Full loop: /init baseline → generate → test → shrink → repeat → viewer
- **`generate_eval_viewer.py`** — Generate self-contained HTML viewer from workspace results

### Existing infrastructure
- **`install_deps.sh`** — Auto-detect platform, install build deps
- **`strip_build_hints.sh`** — One-time: strip in-tree docs from repo, commit so worktrees inherit clean state
- **`setup_worktrees.sh`** — Create isolated git worktrees per trial, write metadata to results/
- **`teardown_worktrees.sh`** — Clean up worktrees (results/ is untouched)
- **`run_graders.sh`** — Deterministic graders, outputs `grading.json` to results dir
- **`display_metrics.py`** — Read CSV, print comparison tables
- **`aggregate_benchmark.py`** — Generate `benchmark.json` from iteration data

### Removed
- **`record_metrics.sh`** — Replaced by `extract_metrics.py`. The old script took 16 positional arguments hand-extracted by the evaluator. Now metrics come deterministically from JSONL transcripts.

## Workspace Organization

Worktrees (ephemeral execution environments) and results (persistent artifacts) are stored separately. Worktrees can be torn down after trials without losing any data.

```
workspace/
├── build_env.sh                   # Created by install_deps.sh
├── history.json                   # Iteration progression
├── repo/                          # Source repo (shallow clone)
├── eval-suites/                   # Eval suite YAMLs
├── agents-md/
│   ├── init-baseline.md           # Generated by /init (the baseline)
│   ├── improved-iter1.md          # Generated from baseline transcripts
│   ├── improved-iter2.md          # Hill-climbed (smaller)
│   └── improved-iter3.md          # Hill-climbed (even smaller)
├── worktrees/                     # EPHEMERAL — torn down after trials
│   ├── iteration-0/
│   │   ├── baseline-t1/           # Self-contained: source + AGENTS.md + build_env.sh
│   │   ├── baseline-t2/

│   └── iteration-1/
│       ├── improved-t1/
```

## Metrics

Extracted deterministically from `transcript.jsonl` by `extract_metrics.py`:

| Metric | Source | Notes |
|--------|--------|-------|
| `n_turns` | result.num_turns | API turns |
| `n_tool_calls` | count of tool_use blocks | Exact from transcript |
| `total_tokens` | modelUsage sum | input + output + cache |
| `total_cost_usd` | result.total_cost_usd | Exact from API |
| `duration_ms` | result.duration_ms | Wall clock |
| `stop_reason` | result.stop_reason | end_turn, timeout, etc. |
| `n_tool_errors` | count of is_error tool_results | Error rate |

Primary eval metric: **pass@1** (did the agent succeed on first attempt?)
Primary efficiency metric: **total_tokens** (cost proxy — fewer tokens = cheaper)

## Key Principles

1. **Transcript is truth.** All metrics come from `transcript.jsonl`. No manual extraction, no estimation.
2. **AGENTS.md is generated, never hand-written.** Sonnet analyzes where Haiku struggled and produces targeted guidance.
3. **Baseline uses /init.** The baseline AGENTS.md is generated by `claude /init` — the standard auto-generated file. Hill-climbed versions must beat this.
4. **Smaller is better.** Track `agents_md_size_words`. If it grows, something is wrong.
5. **Results are persistent.** Trial artifacts live in `workspace/results/`, separate from ephemeral worktrees.
6. **Three roles, three models.** Evaluator (scripts), Generator (Sonnet), Agent (Haiku). Never conflate them.
